## Goal

Set up a queue consumer that retries on transient failures (e.g. an external endpoint being down) and automatically routes jobs to a dead letter queue (DLQ) when all retries are exhausted.

## Steps

### 1. Register the external endpoint as an HTTP-invoked function

{/* TODO: Check if this assertion on non-2xx status calls is correct
AND if there are any changes to allow for other statuses to pass (like 300 or user specified) */}

Register the payment API as an HTTP-invoked function. The engine makes the HTTP call — when the
endpoint is down or returns a non-2xx status, the engine marks the invocation as failed. If this
function is triggered by a queue trigger, the queue worker retries it based on `queue_config`.

<Tabs groupId="language" persist items={["Node / TypeScript", "Python", "Rust"]}>
<Tab value="Node / TypeScript">
```typescript title="payment-processor.ts"
import { init } from 'iii-sdk'

const iii = init(process.env.III_BRIDGE_URL ?? 'ws://localhost:49134')

iii.registerFunction(
  { id: 'payments::charge' },
  {
    url: 'https://api.payments.example.com/charge',
    method: 'POST',
    timeout_ms: 5000,
  },
)
```
</Tab>
<Tab value="Python">
```python title="payment_processor.py"
import os
from iii import III, HttpInvocationConfig

iii = III(os.environ.get('III_BRIDGE_URL', 'ws://localhost:49134'))

iii.register_function(
    "payments::charge",
    HttpInvocationConfig(
        url="https://api.payments.example.com/charge",
        method="POST",
        timeout_ms=5000,
    ),
)
```
</Tab>
<Tab value="Rust">
```rust title="payment_processor.rs"
use iii_sdk::{III, HttpInvocationConfig, HttpMethod};
use std::collections::HashMap;

let iii = III::new(&std::env::var("III_BRIDGE_URL").unwrap_or_else(|_| "ws://127.0.0.1:49134".to_string()));

iii.register_function("payments::charge", HttpInvocationConfig {
    url: "https://api.payments.example.com/charge".to_string(),
    method: HttpMethod::Post,
    timeout_ms: Some(5000),
    headers: HashMap::new(),
    auth: None,
});
```
</Tab>

### 2. Subscribe with retry configuration

{/* TODO: Update this once the new way to specify queues is implemented
(ie. queue: "queue-name" & queues defined in iii's config) */}

Bind the function to a queue topic and configure retry behavior through `queue_config`. When the external endpoint fails, the engine retries with exponential backoff until `maxRetries` is exhausted — then the job moves to the DLQ.

<Tabs groupId="language" persist items={["Node / TypeScript", "Python", "Rust"]}>
<Tab value="Node / TypeScript">
```typescript title="payment-trigger.ts"
iii.registerTrigger({
  type: 'queue',
  function_id: 'payments::charge',
  config: {
    topic: 'order.created',
    queue_config: {
      maxRetries: 5,
      backoffType: 'exponential',
      backoffDelayMs: 2000,
    },
  },
})
```
</Tab>
<Tab value="Python">
```python title="payment_trigger.py"
iii.register_trigger(
    type="queue",
    function_id="payments::charge",
    config={
        "topic": "order.created",
        "queue_config": {
            "maxRetries": 5,
            "backoffType": "exponential",
            "backoffDelayMs": 2000,
        },
    },
)
```
</Tab>
<Tab value="Rust">
```rust title="payment_trigger.rs"
iii.register_trigger("queue", "payments::charge", json!({
    "topic": "order.created",
    "queue_config": {
        "maxRetries": 5,
        "backoffType": "exponential",
        "backoffDelayMs": 2000
    }
}))?;
```
</Tab>
</Tabs>

With this configuration, a failing job follows this timeline:

| Attempt | Delay before retry |
|---------|--------------------|
| 1       | 2 s                |
| 2       | 4 s                |
| 3       | 8 s                |
| 4       | 16 s               |
| 5       | — moved to DLQ     |

### 3. What happens when a job lands in the DLQ

When the payment endpoint is down and all 5 retries exhaust, the engine:

1. Removes the job from the active queue
2. Stores it in the DLQ with the original payload, the last error, and a `failed_at` timestamp
3. Logs a warning:

```
WARN queue="order.created" job_id="..." attempts=5 "Job exhausted, moved to DLQ"
```

The job stays in the DLQ until the engine redrives it. No other jobs on the topic are blocked — processing continues normally for new messages.

### 4. Queue configuration reference

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `maxRetries` | `number` | 3 | Maximum delivery attempts before moving to DLQ |
| `backoffType` | `string` | `"exponential"` | Backoff strategy (`exponential`) |
| `backoffDelayMs` | `number` | 1000 | Base delay in ms between retries |
| `concurrency` | `number` | 10 | Max concurrent jobs for this subscriber |
| `type` | `string` | `"concurrent"` | Queue mode: `"concurrent"` or `"fifo"` |

## Result

Failed jobs retry automatically with exponential backoff. After all retries exhaust, the job moves to the DLQ where it is preserved with its full payload and error context. The engine continues processing new messages on the topic without interruption.

<Callout title="DLQ adapter support" type="info">
  DLQ is fully supported by the **Builtin** and **RabbitMQ** queue adapters. The Redis pub/sub adapter does not support DLQ operations.
</Callout>
